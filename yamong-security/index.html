게으른 대학생

프로젝트 개요​

  • 핵심 목표: 단순 개별 알람(Alert)의 나열이 아니라, 연관된 이벤트들을 하나로 묶어(Case/Incident 단위) 공격의 전체 맥락(Context) 을 보여주는 클라우드 런타임 보안 플랫폼
  • 알림 기능: Discord, Slack, Email, Webhook 등 다양한 채널로 알림을 라우팅하여 클라이언트에게 상황을 전파

작동 과정​

  1. 데이터 수집 (클라이언트)
    • Agent(muninn-agent) 는 eBPF를 이용해 노드 내의 파일/네트워크/프로세스 이벤트를 모니터링
    • Controller(huginn-controller) 는 쿠버네티스 전체의 메타데이터(K8s 오브젝트 변경 등)를 모니터링
    • 두 서비스는 수집한 원시 이벤트 스트림을 중앙 서버의 Kafka로 전송
  2. 데이터 분석 및 처리 (중앙 서버)
    • Detector(bifrost-detector)의 역할
      • Agent가 보낸 런타임 이벤트를 소비하여, 기존에 학습된 정상 패턴(Baseline)과 다른 새로운 이상 행위(Novelty) 를 판별해 Finding 이라는 이름으로 Kafka(finding stream)에 발행
      • 나중에 활용할 수 있도록 정규화된 팩트(Fact Store) DB에 저장
    • Analyzer(mimir-analyzer)의 역할 (맥락 생성 과정)
      • Analyzer는 Detector가 보낸 Finding을 트리거로 삼아 작동
      • Finding이 들어오면, Controller가 저장해둔 K8s 오브젝트 맥락(Cluster Context) 정보와 Detector가 저장해둔 과거 이력/팩트 정보(L2 Context Layer)를 DB에서 함께 조회
      • 이를 결합하여 단순한 Finding을 넘어 다대다 연관성이 묶인 Case-events (침해 의심 묶음/상황)를 생성
    • Dispatcher(heimdall-dispatcher) 및 Dashboard 서빙
      • Dispatcher는 원시 Finding을 직접 처리하지 않고, Analyzer가 종합적으로 판단하여 발행한 최종 결과물인 Case-events 중 알람 등급(Alert-grade)에 해당하는 것들만 소비, 그 후 규칙에 따라 클라이언트(Discord, Email 등)에 라우팅
      • Back API는 이 결과를 받아 대시보드(화면)에 맥락이 풍부한 UI를 서빙

서비스​

Agent​

역할​

각 Kubernetes Worker Node에 배포되어 런타임 신호를 수집하고, 즉각적으로 컨텍스트를 보강하여 중앙으로 전달하는 Node 작업을 책임지는 Node-Local Producer

기능​

  • eBPF 기반 런타임 이벤트 수집
    • 커널 수준의 eBPF 훅을 이용해 각 노드 파드(Pod) 및 컨테이너에서 발생하는 네트워크(Network), 파일(File), 프로세스(Process - fork, exec, exit) 이벤트를 실시간으로 수집
  • Node-local Enrichment (발생 즉시 컨텍스트 보강)
    • 원시 이벤트만 중앙으로 보내는 것이 아니라, 신호가 발생한 노드 안에서 발생 즉시 NAT/Service VIP 해석, Pod IP 매핑, Cgroup 기반 신원(Identity) 확인 등을 수행
    • 쿠버네티스의 IP 재사용이나 NAT에 의한 목적지 왜곡으로 인해 나중에 중앙에서 해석할 때 발생할 수 있는 데이터 정합성 문제를 원천 차단
  • Kafka를 통한 런타임 이벤트 스트림 발행 (Upstream 역할)
    • 보강이 완료된 이벤트를 Kafka의 Runtime topics로 발행

작동 과정​

Phase 1. 초기화 및 환경 탐지 (Bootstrap)

  1. Agent 실행 및 설정 로드: 구동 시 가장 먼저 설정값을 읽고, 현재 속해있는 쿠버네티스 클러스터의 API와 통신하기 위한 클라이언트를 준비
  2. 런타임 식별: 현재 구동 중인 워커 노드의 컨테이너 구동 환경(예: containerd) 위치를 자동으로 탐지하고 접속을 맺음

Phase 2. 컨텍스트 동기화 및 캐싱 (Context Watch & Sync)

eBPF 신호가 뜨자마자 즉시 "누구의 행위인지" 알기 위해 미리 족보를 준비해두는 단계

  1. K8s 감시: Update Manager 모듈이 쿠버네티스 노드 내의 Pod 생성/소멸, Service 변동, 정책 변동 정보 등을 실시간으로 감시(Watch)
  2. 인메모리 캐시 구축: 이 정보들을 바탕으로 초고속 조회를 위한 자체 메모리 캐시인 Cgroup Cache (컨테이너/파드 신원용)와 IP Cache (IP 소유권용)를 구축

Phase 3. 원시 신호 수집 (eBPF Capture)

  1. 커널 훅 삽입: 모든 준비가 끝나면, eBPF Manager가 리눅스 커널 층 깊숙이 eBPF 훅(Hook)들을 삽입(Attach)
  2. Ring Buffer 수신: 노드 전체 컨테이너들에서 발생하는 네트워크 통신, 파일 조작, 프로세스 실행(fork/exec) 원시 기록(Raw event)들을 Ring Buffer로 빨아들임

Phase 4. 발생 즉시 컨텍스트 보강 (Node-local Enrichment)

  1. 디코딩 및 필터링: 빨아들인 원시 이벤트를 해독(Decode)하고, 불필요한 노이즈(예: 루프백 IP 통신, 시스템 기본 파일 읽기 등)는 바로 버림(Drop)
  2. 정보 결합 (Join): 유효한 이벤트의 IP나 PID 같은 단서를 가지고, 앞서 'Phase 2'에서 준비해 둔 캐시(Cgroup / IP)를 즉각 뒤져서 신분(Identity)을 맞춰봄
  3. 맥락 덧씌우기: 이 결합을 통해 "단순 접속 IP"가 "결국 NAT를 거쳐 도달하고자 했던 실제 Service나 백엔드 파드(via 구조체)"로 복원되고, "단순 PID"가 "어떤 네임스페이스의 어떤 파드 소속(WorkloadMeta)" 인지 밝혀짐

Phase 5. 외부 발행 (Kafka Output Publish)

  1. 메시지 봉투 조립: 최종적으로 풍부한 상황 단서가 씌워진 DecodedEvent(행위)를 OutEvent라는 봉투에 담아 WorkloadMeta(신원/출처)와 결합
  2. 토픽별 비동기 전송: 결합된 메시지들을 이벤트 목적에 맞게 Kafka의 3가지 토픽(network-events, file-events, process-events)으로 비동기 전송(Publish)

Controller​

역할​

Kubernetes Object 변화 이벤트를 수집하여 권위 있는(Authoritative) 클러스터 맥락(Context), 출처(Provenance), 지형도(Topology) 이벤트 스트림으로 전송하고, 하위 계층(Downstream)이 이를 Agent의 런타임 이벤트와 조합해 해석할 수 있게 해주는 Cluster Context Event Producer

기능​

  • 워크로드 신원 및 소유권 증명 (Workload Identity & Ownership)
    • Pod, Owner chain, Namespace, Service Account를 기반으로 워크로드 신원 및 소유권 그래프 생성
    • 개별 컨테이너가 실제로 어떤 이미지(image_ref, image_id)로 실행 중인지 명확히 증명
  • 스토리지 및 볼륨 출처 추적 (Storage & Volume Provenance)
    • Agent가 보고한 로컬 파일 접근 경로를 분석하여, 해당 파일이 ConfigMap, Secret, HostPath, PVC(영구볼륨), EmptyDir 중 어느 Kubernetes Source에서 기인했는지 출처(Provenance) 도출
  • 서비스 및 엔드포인트 연결망 구축 (Service & Endpoint Topology)
    • Service 및 EndpointSlice 변경 이벤트를 수집하여 서비스 그래프(Service Graph)와 워크로드 연결망(Connectivity) 구성
    • 개별 파드가 어떤 논리적 서비스 및 엔드포인트 토폴로지에 소속되어 있는지에 대한 환경적 맥락 제공
  • 클러스터 자산 인벤토리 관리 및 영향 범위 산정 지원
    • 생산된 소유권, 저장소, 네트워크 연결망 데이터를 토대로 향후 분석 계층(Analyzer)에서 보안 사고 발생 시 '잠재적 영향 범위(Blast Radius)'를 계산할 수 있는 기반 인프라 데이터 공급

작동 과정​

Phase 1. 클러스터 환경 모니터링 (Context Watch & Sync)

  • API 서버 감시 (Primary): Kubernetes 공식 클라이언트(client-go)의 Informer 메커니즘을 활용하여 API 서버에서 발생하는 Pod, Service, Node 등의 메타데이터 변화를 실시간으로 구독 및 감시(Watch)
  • 런타임 소켓 연결 (Secondary): Node 로컬 영역에 구동 중인 컨테이너 런타임 환경(CRI Endpoint, 예: containerd 소켓)과 연결하여 K8s API만으로는 부족한 컨테이너 런타임 수준의 세부 메타데이터 확보 준비

Phase 2. 족보 및 지형도 구성 (Context Graph Generation)

  • 맥락 정보 식별: 수집 및 감시된 K8s Object 이벤트를 바탕으로 현 워크로드의 소유주(Owner Reference), 결합된 격리 볼륨, 연동된 서비스 자산 관계 등을 정밀 식별
  • 전역 메타데이터 그래프 형태 조립: 단순 변화 알림을 넘어선, 클러스터 내 각종 리소스들이 논리적으로 어떻게 얽혀 있는지 파악이 가능한 '토폴로지 그래프' 형태로 데이터 객체화 수행

Phase 3. 분석 파이프라인 전송 (Context Stream Output)

  • 맥락 데이터 스트림 발행: 조합 완료된 클러스터 단위의 전역 Context 이벤트를 다운스트림 아키텍처(Kafka Stream 등)로 발행(Publish)
  • 분석용 자산(Lookup Surface) 유지: Agent가 보내는 '현장 발생 행위(Event-time Fact)' 파일과 융합(Join)되어 탐지기(Detector) 및 분석기(Analyzer)가 의미 있는 침해 판단을 내릴 수 있도록 클러스터의 '상태 기억 장치(Context Memory)' 역할 수행

Detector​

역할 ​

Agent의 런타임 행위와 Controller의 족보(맥락)를 조합해 정상적인 패턴(Baseline)을 학습하고, 이를 벗어나는 '새로운 행위(Novelty)'나 '알려진 위협(Known-bad)'을 식별하여 **"수상한 단서(Finding)"**를 짚어내는 전위 탐지 엔진 (First-line Anomaly Detector)

기능​

  • 정상 행위 학습 및 베이스라인 프로파일링 (Baseline Generation)
    • 새로운 워크로드(동일한 특성을 가진 파드 그룹)가 배포되면, 의심하기 전 일시적으로 Learning Mode(학습 모드)에 진입
    • 해당 애플리케이션의 정상적인 통신 대상, 주로 실행되는 프로세스 등의 지문(Fingerprint) 데이터베이스를 수집하고 기준선(Baseline)을 스스로 구축
  • 이상 행위(Novelty) 및 알려진 위협(Known-bad) 적발 (Detection)
    • 학습 기간이 끝나고 Securing Mode(보안/감시 모드)로 넘어갔을 때, 평소 기준선에 없던 **완전히 낯선 이상 행위(Novelty)**가 발생하거나, 이미 룰에 박혀있는 명백한 **악성 행위(Known-bad)**가 식별되면 즉각 잡아냄
  • 탐지 단서(Finding) 및 정규화 팩트 발행 (Output)
    • 적발해 낸 이상 징후를 단일 탐지 결과(Finding) 라는 캡슐로 만들어 Kafka의 finding stream으로 발행 (후방의 Analyzer를 깨우는 방아쇠 역할)
    • 차후 Analyzer가 수사를 진행할 때 든든한 증거 자료로 쓸 수 있도록, 탐지 건뿐만 아니라 일상적인 팩트 기록(Normalized Fact)들도 분류하여 보관소(Kafka/DB) 측으로 함께 쏴둠

작동 과정​

Phase 1. 정보 수신 및 문맥 조립 (Ingest & Context Join)

  • 런타임 이벤트 수신: Agent가 Node-local 영역에서 1차 가공하여 보낸 파이프라인 데이터(프로세스, 파일, 네트워크 런타임 이벤트)를 지속적으로 구독 및 수신
  • 맥락 정보 교차 검증 (Join): 수신된 이벤트에 포함된 workload_identity_key 및 root_owner_uid를 Controller가 제공하는 클러스터 지형도(Materialization DB)와 결합하여, 이 이벤트가 정확히 어느 워크로드의 어떤 컨텍스트 하에서 일어났는지 최종 식별

Phase 2. 정상 행위 학습 및 프로파일링 (Baseline Learning)

  • 지문(Fingerprint) 데이터 수집: 새롭게 배포되거나 관찰되는 워크로드가 발견되면, 일정 기간 동안 Learning Mode(학습 모드)에 진입하여 해당 워크로드 전용 proc_info_map(프로세스 실행 정보 등)을 수집
  • 베이스라인(Baseline) 확립: 수집 기간 동안 나타난 정상적인 네트워크 접속지, 주로 쓰이는 프로세스 커맨드 등의 허용 범위와 패턴을 분석하여 해당 애플리케이션만의 고유한 행위 기준선(Baseline) 데이터베이스를 구축

Phase 3. 이상 행위 및 위협 탐지 (Novelty & Anomaly Detection)

  • 보안/감시 모드(Securing Mode) 적용: 학습 기간이 만료되어 정상 프로필 구축이 완료된 워크로드는 즉각 실시간 탐지 대상(Securing Mode)으로 상태가 전환됨
  • 실시간 적발 수행: 이후 유입되는 모든 런타임 이벤트를 구축된 지문 DB(Fingerprint)와 실시간으로 대조(Compare)함. 평소 기준선에 존재하지 않던 아예 낯선 행위(Novelty) 가 발생하거나, 사전에 블랙리스트 된 알려진 악성 패턴(Known-bad) 이 일치할 경우 이를 즉각 이상 징후로 포착

Phase 4. 단서 후방 이관 및 팩트 보존 (Finding & Fact Publish)

  • 탐지 스트림(Finding Stream) 발행: 포착된 이상 징후를 바탕으로 스스로 사건의 전모(Case)를 확정 짓는 권한 행사 없이, "특정 워크로드의 기준선을 벗어난 명백한 단서(Finding)"만을 캡슐화하여 Kafka 탐지 전용 스트림으로 발행 (다운스트림인 Analyzer의 분석 트리거로 활용)
  • 정규화 팩트(Normalized Fact) 기록: 탐지된 이벤트뿐만 아니라, 향후 분석기(Analyzer)가 사건 관계를 엮거나(Correlation) 역추적(Backtrack)할 때 즉시 조회할 수 있도록 주요 궤적들을 Normalized Fact 형태로 DB 및 Kafka에 함께 보존/적재

Normalized Fact Materialization​

탐지 결과와 무관하게 추후 Analyzer가 침해 시나리오를 엮을 증거 자료로 쓰기 위해 normalized fact stream으로 보내는 3가지 양식의 기록 데이터

  • Payload A: 논리 팩트 - 프로세스 (normalized_process_fact): 주로 같은 프로세스 체인(부모 자식 관계)을 파악하기 위해 쓰이는 데이터
필드설명
source_event_id해당 팩트를 기록하게 만든 원본 이벤트 고유 ID
correlation_process_key[핵심] 흩어진 프로세스들을 연결하기 위한 상관관계 공통 키 (Analyzer용 조립 핀)
parent/root lineage ref프로세스가 어디서부터 파생되어 내려왔는지 역추적하기 위한 족보 참조값
실행 문맥프로세스 ID, 실행 명령어 등 상세 행위 사실
  • Payload B: 논리 팩트 - 파일 (normalized_file_fact): 해커가 민감한 탈취나 변조를 시도했는지(직접적 침투 영향) 파악하기 위한 데이터
필드설명

  • Payload C: 논리 팩트 - 네트워크 (normalized_network_fact): 침해가 다른 내부 시스템망이나 외부 인터넷으로 어떻게 전파되었는지(피해 파급 반경 계산) 파악하기 위한 데이터

Analyzer​

역할​

탐지기(Detector)에게 파편화된 '탐지 단서(Finding)'들을 받아, 개별 알람이 아닌 설명 가능한 '하나의 침해 사건(Case)'으로 조직화하는 중앙 분석 및 상관관계 추론 엔진

기능​

  • 사건의 병합 및 연결 (Case Stitching)
    • 산발적으로 터지는 알림들을 개별 건으로 처리하지 않고, 맥락을 추적하여 동일범의 연장선이면 하나의 '사건(Case)' 파일로 병합(Correlation/Stitching)
  • 근원지 및 실제 파급력 추적 (Origin & Observed Impact)
    • 과거의 기록(Normalized Fact)과 클러스터 지형도(Context DB)를 뒤져 수사망을 전개함
    • 공격자가 최초로 뚫고 들어온 시작점(Origin) 이 어디인지 역추적하고, 현재까지 털린 자산과 네트워크가 어디인지 실제 피해 규모(Impact) 를 계산
  • 우선순위 및 에스컬레이션 판단 (Priority & Escalation)
    • 취합된 침해 시나리오를 바탕으로, 치명적인 상황인지 '위험도 우선순위(Priority)' 평가
    • 기준치를 넘기면 상위 책임자에게 통보(Escalation)하기로 결정
  • 최종 사건 보고서 발행 (Case-Events Output)
    • 모든 퍼즐 맞추기가 끝난 후, 탐지 결과와 분석 결과를 담은 결론 보고서를 case-events 스트림으로 발행

작동 과정​

Phase 1. 탐지 정보 유입 및 식별 (Finding Ingest & Context Resolution)

  • 탐지 스트림(Finding Stream) 구독: 앞단에 위치한 Detector가 발행한 각종 이상 행위(Novelty) 및 위협(Known-bad) 단편 이벤트 스트림을 Kafka를 통해 실시간으로 수신
  • 근본 대상 식별 (Context Resolution): 수신된 단서에 포함된 workload_identity_key를 활용하여, 해당 경보가 어떤 클러스터, 네임스페이스, 그리고 구체적으로 어떤 워크로드 오너(Deployment 등) 하위에서 발생한 것인지 침해 대상을 명확히 식별

Phase 2. 과거 행적 및 지형도 교차 조회 (Fact & History Lookup)

  • 정규화 팩트 기반 역추적 (Lineage Track): 발견된 단서(Finding)의 인과관계를 알기 위해, 해당 프로세스의 최초 실행자나 침해 전/후의 상세 네트워크 접속 사실들을 Normalized Fact Store에서 깊이 있게 조회(Lookup)
  • 클러스터 지형도 결합 (Asset Topology Join): Controller 계층이 제공하는 자산 인벤토리 및 서비스 연결망 데이터를 융합. 이를 통해 침해당한 파드가 어떤 쿠버네티스 권한(Service Account)을 쥐고 있는지, 어떤 격리 볼륨(Secret 등)에 접근할 수 있는지 입체적인 구조적 맹점을 파악

Phase 3. 사건 병합 및 타격 범위 평가 (Case Stitching & Impact Analysis)

  • 단일 사건 병합 (Case Stitching/Correlation): 기존에 열려 있는 활성 사건(Active Case) 중 시간적(Time-window), 공간적(Topology), 혹은 프로세스 족보상 연관성이 있는 사건이 존재하는지 판별. 연장선에 있는 위협일 경우 수백 개의 개별 경보로 뿌리지 않고, 기존 사건 스레드(Case ID) 하위에 새로운 증거물로 묶음(Stitch)
  • 근원지(Origin) 및 유효 피해 규모(Observed Impact) 도출: 융합된 팩트들을 기반으로 공격이 최초로 유입된 타점(Origin)을 역추적하고, 내부 시스템 통신 발생이나 보호 파일 접근 등 구체적인 침투 행위로 인해 실질적으로 어디까지 인프라 피해가 닿았는지(Impact)를 논리적으로 계산 및 정량화

Phase 4. 확정 시나리오 발행 및 에스컬레이션 (Case-events Output & Escalation)

  • 위험도 및 우선순위(Priority) 산정: 조합된 실질 피해 사실(Observed Impact)과 위협 수준을 종합적으로 합산하여, 보안 관제 센터에서 당장 확인해야 하는 위협인지에 대한 최종 등급(Severity/Priority) 산정
  • 최종 사건 페이로드 발행 (Case-events Publish): 모든 인과관계 병합과 설명이 끝난 완료된 묶음 데이터를 case-events 형식의 최종 Kafka 토픽으로 발행.
  • 승격 통보 (Escalation): 개입이 즉각 필요한 치명적 위협일 경우 에스컬레이션 대상 플래그를 세우며, 이 스트림을 수신한 Dispatcher(배달 런타임) 가 해당 사건을 최종적으로 보안 운영팀의 Slack이나 이메일 등으로 쏘아 올림

외부 서비스간 통신​

Analyzer가 모든 수사 및 사건 병합을 마치고 외부 서비스(Dispatcher, Back API 등)로 배포하는 최종 보고서 데이터 구조

필드설명
analysis_id현재 진행된 분석 결과의 고유 식별자
finding_id이 분석을 촉발한 원본 탐지 단서(Finding)의 고유 식별자
source_event_id적발의 베이스가 된 최초 원천 이벤트 추적용 키

case_id이 알람 조각이 최종적으로 병합되어 하나로 묶인 통합 사건(Case)의 번호
case_event_type다운스트림 서비스가 읽을 수 있는 공식 알람 타입 (예: 새 사건 생성, 위험도 승격 등)

service_family_key운영팀을 찾기 위한 라우팅 전용 오너십 그룹 식별자
workload_identity_key구체적인 주인을 찾을 때 서브로 쓰이는 워크로드 식별자

case_action시스템이 취한 병합 행동 (예: 새로운 사건 생성함, 기존 사건에 병합시킴)
origin_status공격이 최초 유입된 시작점(Origin)이 완전히 확정되었는지 판별 상태값
origin_reason위 시작점 상태를 입증하는 논리적 이유
analysis_baseline_scope수사관이 어느 범위(컨테이너, 파드, 족보)까지를 '낯선 행위'로 한정 지었는지에 대한 기준

secondary_tags[]관련된 보조 보안 위협 프레임워크 태그들의 목록
priority_score방어 운영팀 관점에서 처리해야 할 우선순위를 수치화 한 점수값
escalation_required[핵심 트리거] true일 경우 보안 담당자의 메신저/페이저 알림을 실제로 울리라는 명령어
escalation_reason알람을 울려야 하는 구체적 이유 (예: "핵심 권한 탈취 정황이 강력함")

Dispatcher​

역할​

파이프라인(Analyzer)에서 확정 지은 침해 사건 정보(Case-events)를 수신하여, 대상 워크로드의 소유권을 판별(Ownership Routing)하고 각 기업의 외부 알림 채널(Slack, Discord, Email 등)에 맞는 UI 템플릿으로 렌더링(Rendering)하여 정확한 담당자에게 통보하는 알림 배포 전담 런타임 (Alert Delivery Runtime)

기능​

  • 메시지 구독 및 알림 대상 판별 (Kafka Consumer & Validation)
    • Analyzer가 발행하는 case-events 스트림을 지속적으로 구독하며, 메시지 중 알림 발송이 명시적으로 허가된 데이터(escalation_required = true)만을 필터링
  • 케이스 중심 소유권 라우팅 (Case-centric Routing)
    • 침해 당한 파드의 cluster_id 및 service_family_key (예: production 네임스페이스의 payment-api 부서)를 추출하여, 사내 조직도의 어떤 팀 슬랙(Slack) 채널로 보낼지 수신처 주소 매핑
  • 맞춤형 UI 템플릿 렌더링 (Template Rendering)
    • JSON 형태의 분석 데이터를 사람이 읽기 좋은 경고문 디자인으로 포장
    • 전송 대상 플랫폼(Provider: Slack, Discord, Email 등)의 규격에 맞춰 DB 내 저장된 템플릿 구조를 불러와 디자인을 입히는(Formatting) 기능 수행
  • 전송 신뢰성 및 멱등성(Idempotency) 보장
    • 일시적 네트워크 단절 등으로 인해 알람이 실패하더라도 재시도하며, 장애가 지속되면 DLQ (Dead Letter Queue, 에러 보관함) 스토리지에 안전하게 우회 보관
    • 알람이 네트워크 재전송 이슈로 인해 동일한 침해 건이 2~3번 전송되는 중복 발송을 원천 차단(멱등성 보장)

작동 과정​

Phase 1. 이벤트 수신 및 필수 요건 검증 (Consume & Validate)

  • 알람 등급 판별 필터링: Kafka의 case-events 토픽에서 흘러들어오는 데이터들을 소비(Consume)하되, Analyzer가 위협으로 확정 지은 escalation_required = true인 알람 데이터만을 메모리에 적재
  • 페이로드 정규화: 수신된 메시지를 발송 가능한 형태의 규격(CaseEventPayload)으로 안전하게 변환하고 필수 데이터의 누락 여부를 1차 검증

Phase 2. 수신처 매핑 및 오너십 결정 (Route Resolution)

  • 소유권(Ownership) 기반 라우팅: 메시지 내에 포함된 service_family_key 등의 족보를 해석하여, DB에 등록된 '수신처 담당 그룹(Responsible Group)'과 대조
  • 채널 확인: "이 결제 파드의 담당자는 핀테크 본부이고, 채널은 #alert-fintech 슬랙 채널이다"라는 정확한 목적지(Notification Channel) 매핑 완료

Phase 3. 알림 템플릿 맞춤 렌더링 (Template Rendering)

  • Provider-specific 변환: 슬랙은 Markdown을, 웹훅(Webhook)은 JSON을 좋아하는 것처럼 도착지의 성격(Provider)이 모두 다름. 목적지가 요구하는 인터페이스에 맞게 템플릿 엔진을 가동
  • 조립(Formatting): Analyzer가 적은 범죄 요약(reason_summary), 위험도 점수(priority_score) 등의 변수(Variable)를 템플릿의 빈칸에 채워 넣어 최종 발송용 페이로드를 조립

Phase 4. 최종 플랫폼 송출 및 사후 처리 (Delivery & Idempotency)

  • 중복 발생 통제 (Idempotency): 내부 캐시 혹은 DB를 조회하여, "이 Case ID의 위험 상승(Escalation) 알림은 3분 전에 이미 한 번 쐈다"는 사실이 확인되면 발송을 즉각 취소(Drop)하여 스팸 방지
  • 발송 및 보관 (Delivery & DLQ): 외부 통신(API)을 찔러 최종 발송을 수행하며, Provider 측의 500에러 다운 등으로 실패 시 지정된 Fallback(우회) 절차를 수행. 끝까지 전송에 실패한 메시지는 영구 보존용 DLQ DB에 덤프(Dump)하여 추후 재처리를 대기
