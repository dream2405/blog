Git Config​

설정 파일 우선순위

  1. .git/config: 특정 저장소에만 적용
    • git config --local 옵션 (기본값)
  2. ~/.gitconfig, ~/.config/git/config: 특정 사용자에게만 적용
    • git config --global 옵션
  3. /etc/gitconfig: 시스템의 모든 사용자와 모든 저장소에 적용
    • git config --system 옵션 (관리자 권한 필요)

유용한 옵션

  • 옵션
    • git config --global user.name "Jaehyun Nam"
    • git config --global user.email namjh@dankook.ac.kr
    • git config --global core.editor emacs
    • git config --global init.defaultBranch main
  • 설정 확인
    • git config --list

로컬 저장소 생성​

명령어

  • git init: 로컬 저장소 생성
    • 해당 폴더에 .git 디렉토리 생성
  • git init -b <branch-name>: 브랜치 이름 지정

파일 추가​

Staging Area에 파일 추가

  • Staging Area는 index라고 하기도 함

명령어

  • git add [파일명]
  • git add [파일1] [파일2]
  • git add .: 현재 디렉토리와 그 하위 모든 변경된 파일들을 한 번에 Staging Area에 추가

커밋​

Staging Area에 있는 파일 변경 사항을 로컬 저장소에 저장

명령어

  • git commit -m "변경 내용을 간단히 요약한 메시지"
  • git commit -s -m "변경 내용을 간단히 요약한 메시지": 개발자 서명 규약(DCO)에 동의
  • git commit -am "커밋 메시지": 수정된 파일들을 자동으로 스테이징하고 커밋
    • 깃이 추적하지 않는 파일은 git add로 추가 해야됨!
  • git commit --amend: 기존 커밋을 완전히 새로운 커밋으로 대체
    • 옵션 없음: 에디터를 열어 최근 커밋 메시지 수정
    • git commit --amend -m "메시지": 명령줄에서 커밋 메시지 교체
    • git commit --amend --no-edit: 메시지 수정 없이 스테이징된 변경사항만 최근 커밋에 추가
    • 원격 저장소에 이미 올라간 커밋은 충돌 발생!
      • git push --force로 덮어써야 함

푸시​

로컬 저장소의 커밋 이력을 원격 저장소에 업로드

명령어

  • git push
  • git push -u origin main: 로컬 브랜치와 원격 브랜치를 연결
    • 한 번 설정해두면 이후에는 브랜치 이름을 입력할 필요 없이 git push나 git pull만 입력해도 자동으로 origin의 main과 통신
  • git push --force: 원격 저장소의 히스토리를 로컬 히스토리로 강제로 덮어씀
  • git push --force-with-lease: 내가 마지막으로 확인했던 원격 저장소의 상태와 현재 서버의 상태가 다를 경우 푸시를 거부
  • git push origin -d feature-login: 원격 저장소에 있는 브랜치 삭제
    • git branch -d로는 원격 저장소의 브랜치를 삭제할 수 없음!

태그​

명령어

  • git tag: 저장소에 있는 모든 태그 목록 확인
  • git tag v1.0.0: 특정 커밋에 이름을 붙임
  • git tag -a v1.0.0 -m "첫 번째 정식 출시 버전": 주석 맺은 태그 붙이기
  • git tag -a v0.9.0 a1b2c3d -m "이전 버전 추가 태깅": 특정 커밋에 주석 맺은 태그 붙이기
  • git tag -d v0.9.0: 로컬 저장소에서 태그 삭제
  • git push origin v1.0.0: 원격 저장소에 태그 푸시
  • git push origin --tags: 모든 태그를 한 번에 푸시
  • git push origin --delete v1.0.0: 원격 저장소에서 태그 삭제

상태 확인​

워킹 트리의 상태 확인

  • 커밋할 파일, 추적 안 된 파일, 수정된 파일 구분 표시

명령어

  • git status

로그 확인​

Show commit logs

명령어

  • git log: 가장 최근 커밋부터 역순으로 표시
  • git log --oneline: 각 커밋을 한줄로 요약
  • git log -p: diff도 보여줌
  • git log --oneline --graph --all: 브랜치를 텍스트 그래픽으로 시각화
  • git log -n 5: 최근 5개의 커밋만 확인
  • git log -- README.md: 특정 파일이 변경된 커밋들만 확인

변경사항 확인​

변경 사항을 비교하고 시각화

명령어

  • git diff: 워킹 디렉토리 vs 스테이징 영역
  • git diff --cached: 스테이징 영역 vs 최신 커밋 (HEAD)
  • git diff HEAD: 워킹 디렉토리 vs HEAD
  • git diff <commit_id_A> <commit_id_B>: 두 커밋간 비교
  • git diff branch_A..branch_B: 두 브랜치 간 비교

워킹 디렉토리/스테이징 되돌리기​

명령어

  • git restore <파일명>: 파일을 마지막 커밋 상태로 되돌림
  • git restore --staged <파일명>: git add로 올린 파일을 스테이징에서 제외
  • git restore --source=<커밋해시 또는 브랜치명> <파일명>: 지정한 소스(commit/branch)에 있는 해당 파일의 내용이 현재 워킹 디렉토리에 덮어씌워짐

커밋 취소​

Reset current HEAD to the specified state

HEAD: 현재 작업 중인 브랜치의 가장 최신 커밋을 가리키는 포인터

  • 지금 어디에서 작업 중인지를 알려주는 Git의 위치 표시
  • 브랜치를 이동하면 HEAD도 함께 이동
  • HEAD의 조상 찾기
    • HEAD^<n>: n번째 부모 커밋 선택
    • HEAD~<n>: 첫번째 부모만을 따라서 n번 올라감
    • HEAD^{/<text>}: HEAD 위치에서 갈 수 있는 커밋들 중 커밋 메시지가 <text> 정규 표현식에 매칭되는 제일 최신 커밋

명령어

  • git reset --soft HEAD~1: 최근 커밋만 삭제, 변경 내용은 스테이징 영역에 유지
  • git reset --mixed HEAD~1: 커밋 + 스테이징도 삭제 (기본 옵션), 변경 내용은 워킹 디렉토리에 남김
  • git reset --hard HEAD~1: 모든 내용 삭제 (커밋, 스테이징, 파일 수정까지 모두 제거됨)
    • 되돌릴 수 없음!!

Revert​

Revert some existing commits

명령어

  • git revert <commit>: 지정된 커밋을 되돌리는 커밋을 만듦
  • git revert A..B: A 커밋 이후 B 커밋 이전의 커밋들을 역순으로 되돌리는 커밋을 만듦
  • git revert <해시> --no-edit: 커밋 메시지 수정 없이 커밋을 되돌리는 커밋을 만듦
  • git revert -n <커밋해시>: 커밋을 되돌리고 스테이징 영역에 추가하지 않음
    • git revert --no-commit <커밋해시>와 동일
  • git revert -m 1 <머지커밋해시>: 머지할 당시 내가 머물고 있던 브랜치의 커밋으로 되돌리는 커밋을 만듦
    • git revert로는 머지 커밋을 되돌릴 수 없음

파일 삭제/이동​

git의 스테이징 영역 자동 관리

명령어

  • git rm <파일경로>: 파일 삭제 + 삭제한 상태 스테이징에 반영
    • rm은 삭제 후 따로 git add 필요
  • git rm --cached <파일경로>: 실제 파일은 그대로 두고 스테이징에서 제외
  • git rm -r <디렉토리>: 디렉토리 삭제 + 삭제한 상태 스테이징에 반영
  • git mv <이전경로> <새경로>: 파일 이동(또는 이름 변경) + 그 변화를 스테이징 영역에 즉시 반영
    • git rm과 git add를 동시에 수행

브랜치 확인 및 생성​

List, create, or delete branches

명령어

  • git branch: 현재 로컬 브랜치 확인
    • 현재 위치한 브랜치에 별표(*) 표시
  • git branch -r: 원격 브랜치 확인
  • git branch -a: 모든 브랜치 확인
    • 로컬 브랜치와 원격 브랜치 모두 확인
  • git branch <branch-name>: 브랜치 생성
  • git branch -d <branch-name>: 브랜치 삭제
    • 병합되지 않은 브랜치는 삭제 불가능
  • git branch -D <branch-name>: 강제 삭제
  • git branch -m [old-name] [new-name]: 브랜치의 이름을 변경
    • 병합되지 않은 브랜치의 이름 변경은 불가능
    • 병합된 브랜치의 이름 변경은 가능

브랜치 이동​

Switch branches

명령어

  • git switch <branch-name>: 기존 브랜치로 이동
  • git switch -c <branch-name>: 새로운 브랜치 생성 후 이동
    • git branch <branch-name>과 git switch <branch-name>을 동시에 수행
  • git switch -: 바로 이전에 있던 브랜치로 돌아가기
  • git switch --detach <commit>: 특정 커밋 시점으로 이동 (임시 확인용)

브랜치 병합​

Merge branches

Fast-forward vs Three-way 병합

  • Fast-forward: 병합 대상 브랜치가 병합할 브랜치의 최신 커밋을 이미 가지고 있을 때
    • 병합 대상 브랜치의 HEAD가 병합할 브랜치의 HEAD를 따라가게 됨
  • Three-way: 병합 대상 브랜치가 병합할 브랜치의 최신 커밋을 가지고 있지 않을 때
    • 병합 대상 브랜치의 HEAD가 병합할 브랜치의 HEAD를 따라가게 됨
    • 새로운 병합 커밋 생성

명령어

  • git merge <branch-name> -m "커밋 메시지": 현재 브랜치와 <branch-name> 브랜치를 병합
    • Fast-forward가 가능한 상황이라면 Fast-forward 병합
    • Fast-forward가 불가능한 상황이라면 Three-way 병합
  • git merge --no-ff <branch-name>: Fast-forward가 가능한 상황이라도 강제로 머지 커밋 생성
  • git merge --ff-only <branch-name>: Fast-forward가 가능한 상황만 병합
  • git merge --abort: 병합 중단
  • git merge --continue: 병합 중단 후 병합 재개

충돌 해결

  • 머지 포기
    • git merge --abort로 머지 시도 전 상태로 깨끗하게 되돌림
  • 충돌 직접 해결
    1. 파일 수정: Git이 충돌을 표시한 파일을 열어 내용을 원하는 대로 정리
    2. 스테이징: 수정이 완료된 파일을 git add 명령어로 인덱스에 올림
    3. 마무리: git commit 또는 git merge --continue를 실행하여 머지를 최종 완료
      • git merge --continue는 머지 중이 아닌데 실수로 입력하면 에러 메시지를 띄워주므로 더 안전함

충돌 해결을 돕는 도구들

  • 머지 툴 사용: git mergetool 명령어를 입력하면 그래픽 기반의 머지 도구가 실행되어 시각적으로 코드를 비교하며 해결 가능
  • Diff 확인: git diff를 실행하면 현재 브랜치(HEAD)와 합치려는 브랜치(MERGE_HEAD) 간의 차이점을 3-way diff 형식으로 보여줌
  • 브랜치별 로그 대조: git log --merge -p <경로>를 입력하면 먼저 현재 브랜치의 변경 사항을 보여준 뒤, 이어서 상대 브랜치의 변경 사항을 나란히 보여줌

원본 파일 직접 보기 (Index 번호 활용):

  • git show :1:파일명 – 두 브랜치의 공통 조상(Base) 상태를 보여줌
  • git show :2:파일명 – 현재 내가 위치한 브랜치(HEAD)의 상태를 보여줌
  • git show :3:파일명 – 합치려고 가져온 브랜치(MERGE_HEAD)의 상태를 보여줌
노트

머지 충돌이 발생했을 때, Git 인덱스는 파일을 네 가지 상태로 관리

스테이지 번호명칭의미
0Normal충돌이 없는 평범한 상태 (평소의 모든 파일은 이 상태)
1Common Ancestor두 브랜치가 갈라지기 전의 공통 조상(원본) 버전
2Target (HEAD)현재 내가 있는 브랜치의 수정 버전 (내 코드)
3Source (MERGE_HEAD)합치려고 가져온 브랜치의 수정 버전 (상대방 코드)
# 공통 조상 버전의 내용을 파일로 저장해서 보고 싶을 때
git show :1:README.md > ancestor_version.md

# 상대방(Stage 3)의 내용으로 현재 파일을 완전히 덮어쓰고 싶을 때
git checkout :3:README.md

Stash​

Stash the changes in a dirty working directory away

명령어

  • git stash: 워킹 디렉토리와 스테이징의 변경사항을 stash에 저장하고 HEAD 커밋 상태로 되돌림
    • 추적되지 않는 파일은 워킹 디렉토리에 그대로 남음
  • git stash -u: 추적되지 않는 파일도 stash에 저장
    • 추적되지 않는 파일은 스테이징 영역으로 이동
    • ignore된 파일은 stash에 저장되지 않음
  • git stash -a: 무시된 파일까지 전부 다 stash에 저장
  • git stash push -m "working on login": 메시지를 추가하여 stash에 저장
  • git stash push path/to/file.js -m "이 파일만 잠시 보관": 특정 파일만 stash에 저장
  • git stash list: 스타시 목록 확인
  • git stash apply: 스타시를 적용
  • git stash pop stash@{0}: 특정 인덱스를 지정해서 꺼내기
  • git stash drop stash@{0}: 특정 인덱스를 지정해서 삭제
  • git stash clear: 모든 스타시를 삭제
  • git stash show: 스타시의 변경사항을 확인

리베이스​

Reapply commits on top of another base tip
연쇄적인 충돌 발생 가능성 주의

명령어

  • git rebase -i HEAD~3: 최근 3개 커밋을 수정 - 인터렉티브 리베이스
    • pick: 커밋을 그대로 사용
    • reword: 커밋 메시지만 변경
    • edit: 커밋 내용을 수정
    • squash: 이전 커밋과 하나로 합치기 (지저분한 기록 정리용)
    • drop: 커밋 삭제
  • git rebase --continue: 리베이스 중단 후 리베이스 재개
  • git rebase --abort: 리베이스 중단

Cherry-Pick​

다른 브랜치의 특정 커밋 하나만 선택하여 적용하는 명령어

명령어

  • git cherry-pick <커밋해시>: 특정 커밋을 현재 브랜치에 적용
  • git cherry-pick <해시1> <해시2> <해시3>: 여러 커밋을 순서대로 적용
  • git cherry-pick A..B: A 커밋 이후 B 커밋까지의 커밋들을 역순으로 적용
  • git cherry-pick --continue: 리베이스 중단 후 리베이스 재개
  • git cherry-pick --abort: 리베이스 중단
  • git cherry-pick -n <해시>: 커밋을 적용하지 않고 스테이징 영역으로 이동
  • git cherry-pick -x <해시>: 커밋 메시지 끝에 "(cherry picked from commit ...)" 문구 자동으로 추가
